# Bridge — Telegram-бот Bloom («Мост»)

## Назначение

Рантайм: **aiogram** + опционально **Gemini** / **Groq**, лог диалогов, участники марафона, дайджест мечт из **PostgreSQL**.

Имя ассистента в проекте: **Bridge**. Правило для ассистента: к владельцу проекта — **по-дружески и на «ты»**.

**Bloom** — персонаж и тон общения в Telegram; **Bridge** — имя ассистента/бота в этом репозитории.

**Подпись в переписке с третьими лицами:** любой текст, который Макс пересылает **не ему** (Форж, другой разработчик, ассистент и т.д.), ассистент в этом репозитории оформляет **с явной подписью «Bridge»** в конце или в шапке — чтобы получатель понимал источник и не путал с голосом Макса.

**Принцип решений:** слова «не хочу» в смысле «давай похуже, зато проще» не используем — выбираем **лучший** вариант из реально доступных и двигаемся. Точка.

## Bridge ↔ Island (Остров), MVP

Соцсеть **Остров** (веб, `islanddream.ru`) хранит цели, шаги и расписание; бот Bloom — клиент к HTTP API, без дублирования бизнес-логики расписания в коде бота.

### Что уже согласовано по API (сверка с backend)

- Расписание: `GET /schedule?user_id=…&date_from=…&date_to=…` — элементы с полями `dream_id`, `source_type` (`step` \| `book`), `source_id`, `title`, `date`, `completed`. Неизвестный `source_type` в будущем — логировать и пропускать.
- Обычный шаг: `PATCH /dreams/{dream_id}/steps/{step_id}?user_id=…` с телом, например `{"completed": true|false}`, `{"deleted": true}`, `{"deadline": "YYYY-MM-DD"}` (перенос; по умолчанию от бота можно ставить «завтра»).
- Книга: `POST /dreams/{dream_id}/books/{book_id}/log?user_id=…` с телом минимум `date`, `minutes_spent` — на стороне БД upsert по `(book_id, date)`, повторные нажатия идемпотентны.

### Авторизация и безопасность (текущее состояние → MVP)

Сейчас многие endpoint-ы принимают `user_id` в query **без** bearer/service token (риск подмены `user_id`). Для MVP рекомендуется ограничить доступ к API для машины с ботом (сеть, allowlist IP) и по возможности выставить **`X-Api-Key` на nginx** хотя бы для `/schedule` и update-эндпоинтов; полноценный bot token/JWT в backend — отдельный этап.

### Связка Telegram ↔ пользователь Острова

Целевая колонка: `users.telegram_id` (BIGINT, уникально). Готового «привязать в один клик» в API на момент согласования не было; целевой flow: одноразовый код из ЛК → команда бота `/link CODE` → подтверждение на backend (коды с TTL, одноразовость). Реализация — на стороне Острова + хендлеры в `bridge_bot.py`.

### Антидубли вечернего отчёта

Целевая таблица на стороне Острова, например `daily_report_log`: `user_id`, `report_date`, `source` (`manual` \| `cron`), `sent_at`, опционально `payload`; уникальность `(user_id, report_date)`. Ручной отчёт до дедлайна делает upsert; крон вечером не шлёт повтор, если день уже закрыт.

### Отчёт в общий чат марафона

На первом этапе не делаем; только личка. Сводка в группу — после стабилизации ЛС-сценария.

### Совместная спецификация с Островом (ответ Форжа, 2026)

Общее для обоих концов и **версионируемо в git**: контракт (HTTP, поля, коды ошибок, **кто владелец каких данных**), по возможности снимок схемы БД для полей, которыми пользуются и веб, и бот. Правило процесса: **поменял API — обновил спеку в том же изменении или сразу следующим коммитом.**

**Предпочтительный канон:** отдельный небольшой репозиторий только под документацию (например `island-bridge-contract` / `dream-bridge-handbook`): `README.md`, `api.md`, `db-shared.md`, при желании `runbook.md`. Доступ на запись — Макс, Форж и правки из репозитория `bridge`; мерж или прямые пуши — по договорённости.

**Шаблон файлов** для этого репозитория лежит здесь: **`docs/handbook/`** — после создания пустого репо на GitHub содержимое копируется туда (или делается первый коммит из этой папки). Пошагово: см. `docs/handbook/README.md`.

**Если третий репо на старте не заводим:** канон в репозитории Острова, например `Readme/bridge-integration.md` (или раздел рядом с `tables.md` про поля, которые читает/пишет бот); **в этом README** — одна явная ссылка на актуальные контракты. Плюс: спека в одном PR с изменением `main.py`. Monorepo для вашей раскладки деплоев не обязателен; Notion/Docs — только зеркало, **истина в PR** — в markdown в git.

**Долгосрочно:** дублирование `user_profiles` и БД Острова убрать; единый источник правды для общих пользовательских данных — в БД Острова. Вынесение biography / хвоста диалога / `bloom_traits` — отдельное проектирование поверх той же спеки.

### Реализация в этом репозитории

- Модули **`island_api.py`**, **`island_jobs.py`**, **`island_state.py`**: запросы к API Острова, крон утро/вечер (по `ISLAND_*`), локальные файлы в **`data/`** (привязки `telegram_id → user_id`, учёт отправленного отчёта за день), пока на backend нет `daily_report_log` и готового POST привязки.
- В личке: **`/link`**, **`/report`**, кнопка «Подробнее» после вечернего блока.
- Переменные: **`.env.example`** (`ISLAND_API_BASE_URL`, `ISLAND_SCHEDULE_ENABLED`, и т.д.).

## Принципы

1. **Решения** — не «как попроще», а **лучший вариант** из доступных (см. принцип выше).
2. **Корень** — только то, что нужно для **запуска** бота и общих модулей. Новые фичи бота — рядом с `bridge_bot.py` или отдельным модулем с понятным именем (`dream_db.py`), без свалки скриптов.
3. **`_py_/`** — всё, что запускается руками (список моделей, тестовая отправка в Telegram). Разовое — удалить после использования.
4. **Секреты** — только в `.env` (в git не попадает). Пример — `.env.example`.
5. **Интеграция с Островом** — через HTTP API и переменные окружения (базовый URL, ключи); не подменять `/schedule` собственными правилами в боте.

## Профили и лог диалога (`user_profiles/`)

- **Участники** из `bridge_participants.PARTICIPANTS` — в своей папке (поле `folder`), например `user_profiles/Макс/`.
- **Гости** без записи в карте — отдельная папка `user_profiles/<имя_из_Telegram>_<telegram_id>/` (если уже есть каталог `*_<id>` после миграции, используется он).
- **`chat_history.txt`** — автолог переписки с ботом; хвост в промпте. В `.gitignore`.
- **`biography.txt`** — справка о человеке; при первом контакте создаётся пустым при отсутствии (часто дальше вручную).
- **`bloom_traits.txt`** — краткие наблюдения Блума о человеке (что даётся легко/сложно, чем помочь); создаётся с шаблоном при первом контакте; в `.gitignore`.
- **Конфиденциальность:** данные участников не для публичного оглашения; **владелец бота (Макс)** по запросу получает полную информацию из профилей и логов (заложено в `BLOOM_START_PROMPT.txt` и в коде подстановки файлов для админа).
- При **`/start`** и первом сообщении в личке вызывается `ensure_profile` (папка, `biography.txt`, `bloom_traits.txt`). Повторный «Старт» после удаления чата не создаёт второй профиль при том же `telegram_id`.
- Разовая миграция из `dialogy/`: `_py_/migrate_dialogy_to_chat_history.py`.

## Канон Блума (`Bloom/`)

- Рабочие файлы в **корне `Bloom/`**: `BLOOM_START_PROMPT.txt`, `bloom_biography.txt`, `bloom_body.txt`, `README.md`, `memory_corpus.txt` и т.д. — см. **`Bloom/STRUCTURE.md`**.
- **`Bloom/archive/bloom_2026_snapshot/`** — снимок старого дерева для выжимок; рантайм бота **не** читает архив напрямую.
- **`CURSOR_BOOTSTRAP.md`** — старт в Cursor (не в промпт бота).

Переменная **`BLOOM_HOME`** — абсолютный путь к каталогу с `BLOOM_START_PROMPT.txt` (по умолчанию `.../bridge/Bloom`).

## Дорожная карта (фазы 2+)

| Фаза | Содержание |
|------|------------|
| **1 (текущая)** | Порядок в каноне Блума, актуальный `BLOOM_START_PROMPT`, профили с `bloom_traits.txt`, документация. |
| **2** | Подключение к БД (тестовый контур), синхронизация с `user_profiles/` (источник правды — отдельное решение). |
| **3** | Обучение/правила: отличать отчёты марафона от обычных сообщений в истории чата. |
| **4** | Аналитика: манифесты, привычки, итоги марафона. |
| **5** | Ежедневный цикл (~21:00): рефлексия и одно предложение улучшения → **только владельцу** в личку. |

Шаг за шагом, с ежедневной корректировкой поведения Блума.

## Запуск

```bash
source venv/bin/activate
python bridge_bot.py
# или через systemd: bridge start | restart | status | logs
```

### Автономность (бот не должен зависеть от Cursor)

Бот **не привязан к Cursor** по коду: если он «умирает» при закрытии IDE, обычно запущен **второй экземпляр** из терминала IDE или обрывается **user-сессия** systemd.

1. **Один процесс.** Рабочий вариант — только **systemd user**: `systemctl --user enable --now bridge-bot.service`. Не держи параллельно `python bridge_bot.py` в терминале Cursor — иначе два бота или путаница. Проверка: `pgrep -af bridge_bot.py` (должна быть **одна** строка с `venv/bin/python`).
2. **Юнит в репозитории:** `systemd/user/bridge-bot.service` — копия для установки в `~/.config/systemd/user/` (в юните используется `%h` вместо жёсткого `/home/...`). После правок: `./bridge restart` (скрипт делает `daemon-reload`).
3. **После выхода из системы / конца SSH.** Службы `systemctl --user` по умолчанию останавливаются вместе с сессией пользователя. Чтобы бот жил **без активного входа**, один раз (нужен sudo):

   `sudo loginctl enable-linger $USER`

   Проверка: `loginctl show-user "$USER" -p Linger` → `Linger=yes`.
4. **Альтернатива на сервере:** отдельный system-юнит от root (`/etc/systemd/system/`) с `User=makc` — если не хотите зависеть от user-linger вообще.
